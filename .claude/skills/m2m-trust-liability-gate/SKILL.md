# M2M Trust/Liability Gate — Sovereign Execution Standard

## Constitutional Role

This skill is the authentication checkpoint between creation and distribution.
It does not polish output. It does not rewrite for style.
It asks the binary question that separates authentic sovereign work from
performative professional content — and it does so at every exit point,
without exception.

The gate is the practical expression of Must-Haves 03 (Liability) and 01 (Trust)
in the Seven Must-Haves standard. It fires after drafting, before distribution.

---

## The Binary Gate Question

> **Does this output build TRUST — or does it create LIABILITY?**

There is no middle ground. Every sentence, claim, promise, and positioning
statement in the output lands on one side of this line.

**Trust side** — the output:
- Demonstrates credible, verifiable capability
- Deepens the relationship with the recipient
- Delivers value before asking for anything
- Reflects the Sovereign Architect's authentic domain authority
- Could be signed, presented, and stood behind in a C-suite room

**Liability side** — the output contains:
- Unverified statistics or projections stated as fact
- Overpromised outcomes or guarantees
- Borrowed credibility not earned by M2M or its principals
- Positioning that conflicts with documented capability
- Anything that would require a retraction, clarification, or apology later

**Rule:** If any sentence answers "liability" — that sentence does not ship.
It is rewritten or removed before the gate closes.

## The Seven Must-Haves — Pre-Distribution Checklist

Every M2M output must pass all seven before it exits the platform.

### Quality Layer (05)

**01 — Trust**
> Does this output earn the room before it asks anything of it?
- Leads with demonstrated value, insight, or evidence
- Tone is authoritative without being performative
- The recipient feels served, not sold to
- PASS condition: Yes, this earns entry before making a request

**02 — Context**
> Is it calibrated to this specific reader, room, and distribution channel?
- C-suite output: ROI-first, 3-sentence summary, single decision ask
- Mid-market output: process clarity, timeline, implementation path
- Individual output: narrative authority, empathy, next step named
- PASS condition: The register matches the recipient — not a generic template

**03 — Liability**
> Does anything here expose M2M, the client, or a third party?
Run the binary gate (above) on every claim
- No unverified statistics presented as fact
- No guarantee of outcomes not contractually established
- No third-party attribution without source confirmation
- PASS condition: The Trust/Liability gate closes on the Trust side for all content

**04 — Taste**
> Does it meet the Navy/Gold sovereign aesthetic and C-suite craft standard?
- Typography, spacing, and structure reflect M2M brand system
- Language is precise — no filler phrases, no corporate boilerplate
- The output looks and reads like it came from the highest standard room in the building
- PASS condition: "Always in the cloud of exceptional" — no compromise on craft

**05 — Intent**
> Is the sovereign objective declared — not implied?
- The purpose of the output is explicit, not buried
- The recipient knows what they are being asked to do, think, or decide
- Why before what — the rationale precedes the ask
- PASS condition: Objective is declared within the first 30% of the content

### Doctrine Layer (02)

**06 — Intentional Intent**
> Does the Architect's 20% authenticate and distinguish this output?
- The lived experience, discernment, and domain authority of Dr. Kevin A. Smith
  is felt in the work — not just appended as a signature
- The output could not have been produced by a generic prompt to a generic model
- The sovereign voice is present: directive, warm, prophetic, earned
- PASS condition: Remove the name from this output — can it still only be M2M?

**07 — Fractional Legitimacy**
> Is the execution sourced, the close owned, and the output signed?
- The pipeline (Claude, Make.com, Supabase, automation) executed the 80%
- The Sovereign Architect authenticated the 20%: objective set, output reviewed,
  voice confirmed, deployment authorized
- The output carries the FL/II marker — it is verified, not generic
- PASS condition: "To the work." — can be signed and shipped

## Applying the Gate by Output Type

### Client Deliverable (SOW, Proposal, Report, Assessment)
1. Run binary gate on every quantitative claim
2. Confirm all seven must-haves present
3. Check distribution context: is the format calibrated to this client's C-suite level?
4. Verify Intentional Intent: does the Sovereign Architect's authority distinguish this from a commodity deliverable?
5. Sign off: FL/II authenticated → "To the work."

### Automated Output (Make.com → Gmail, Judge Model, Jotform response)
1. Binary gate on every AI-generated sentence before routing
2. Context check: does the automation know which OS lane (PIVOT/BRIDGE/Human OS) it's serving?
3. Liability check: are there any hallucinated statistics, unverified claims, or blanket promises?
4. Trust check: does the output serve the recipient before asking of them?
5. If any gate fails: route to HUMAN_REQUIRED — do not auto-approve

### Keynote / Speaking Content
1. Every statistic: sourced and verified (HBR, Glassdoor, BLS, peer-reviewed)
2. Every story: authentic — lived experience of principals, not fabricated case studies
3. Taste: cadence, scripture alignment (Isaiah 43:19, Proverbs 20:29), and prophetic register
4. Intent: every slide has one declared purpose — not a summary, a directive
5. Intentional Intent: "When The Ground Moves" — the room should feel the Architect, not just hear a speaker

### Social / Content (LinkedIn, Insights Articles, Platform Posts)
1. Binary gate: does this establish M2M's authority, or does it perform it?
2. Context: is this calibrated to the editorial calendar lane (PIVOT/BRIDGE/Human OS)?
3. Liability: no projection, no guarantee, no borrowed credibility
4. Taste: every post reflects the brand — Navy/Gold, sovereign, non-generic
5. Intent: what action, belief shift, or relationship does this content produce?

### Regulated-Vertical Client Output (Legal / Healthcare / GovCon)
1. Every quantitative claim: sourced or immediately flagged HUMAN_REQUIRED — no softening with "approximately"
2. Tone standardization: output must match the client firm's established communication register — not M2M's sovereign default
3. Precedent consistency: if this output references prior M2M deliverables in the same vertical, language, framing, and positioning must be internally consistent
4. Audit trail: what Claude drafted versus what the Sovereign Architect authenticated is recorded — immutable
5. Gate closes only on the Trust side — liability exposure in regulated verticals is zero-tolerance with no exceptions

### Email / Outreach (Wendell Brown tier, C-suite targets, Partner outreach)
1. Trust: subject line earns open before body asks for time
2. Context: does this email know it is entering a C-suite inbox?
3. Liability: no overpromise in the ask; capability statement is accurate and documented
4. Taste: 30:12 structure (30-word subject, 12-sentence body max) unless SOW/proposal context
5. Intent: one ask per email — the decision point is clear and frictionless

## Self-Check Prompts (Internal — Architect Use)

Before releasing any output, ask aloud or in writing:

1. "Would I present this in the Tuck room to Wendell Brown and stand behind every word?"
2. "If a reporter pulled this output tomorrow, would it represent M2M the way I intend?"
3. "Does the person receiving this feel served — or processed?"
4. "Is my voice in this — or just the model's?"
5. "What is the one thing this output must accomplish? Did it accomplish it?"

## Gate Failure Protocol

If the binary gate fails on any content block:

| Failure type | Action |
|---|---|
| Unverified statistic | Remove or replace with sourced equivalent — never soften with "approximately" |
| Overpromise | Rewrite to reflect documented capability — never hedge with disclaimers |
| Generic voice | Rewrite opening and close with Sovereign Architect register before re-review |
| Context mismatch | Reformat for correct distribution register before re-review |
| Missing Intent declaration | Add sovereign objective statement in first paragraph |
| Taste failure | Return to brand standard — no ship until craft meets C-suite threshold |

**Hard rule:** A failed gate is not a reason to lower the standard.
It is a reason to rebuild the output.
